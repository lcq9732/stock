# 历史行情数据平台 — 设计文档

状态：核心功能已实现（`StockPlatform.Logic/Data/Fetcher/Analyzer`）。本文档记录设计方向，随实现同步更新。

## 1. 背景与目标

曾经有一个 StockAnalyzer（`src/StockAnalyzer.UI`）是实时/近实时的 A 股技术面选股工具，面向"手动添加几只自选股，定时刷新看是否满足7条技术规则"的场景，数据只保留最近90天左右，用于计算指标，不做历史留存。

新的需求是做一个**独立的历史数据平台**：批量获取全市场（或指定范围）近3年的行情与基本面数据，落地保存，供后续开发的分析程序使用。这个平台与当时的 StockAnalyzer 是两条独立的线，通过约定好的 SQLite 数据文件解耦。

**状态变更记录（2026-07-07）**：StockAnalyzer 那套7指标打分逻辑后来被移植成了 [分析程序](analysis-app-design.md) 的"金叉法"（见该文档3.2.2节），确认覆盖完整后，`StockAnalyzer.UI`/`StockAnalyzer.Logic`/`StockAnalyzer.Data` 三个项目已被整体删除（见 analysis-app-design.md 第5节）。本节及下文提到"StockAnalyzer"之处均指这个已删除的历史项目，保留描述是为了说明当时的设计动机，不代表代码仍然存在。

## 2. 总体架构：两个独立程序

```
┌─────────────┐  本地抓取  ┌──────────────┐  手动拷贝  ┌─────────────┐
│ 数据获取程序  │ ────────▶ │ 本地 SQLite   │ ────────▶ │  分析程序     │
│ (Fetcher)   │           │(current.sqlite)│(total.sqlite)│ (Analyzer)  │
└─────────────┘           └──────────────┘           └─────────────┘
```

- **数据获取程序**：独立运行，抓取写入本地 SQLite（`current.sqlite`）。**抓取动作必须由用户手动点击触发，不会自动静默运行**（不挂定时任务自动跑，见6.7）。
- **分析程序**：独立的程序，只读查询用户手动拷贝过去的数据库文件（约定文件名 `total.sqlite`）做分析。
- 两者之间**只通过 SQLite 文件的表结构**耦合，互不依赖对方的代码。

**状态变更记录（2026-07-09）**：最初设计（见第6节）是"本地生成 master/daily 文件 → 手动上传网盘 → 分析程序从网盘拉取合并"，用来支持没有服务器时的多机共享。实际开发下来，分析程序从来没有读取过 master/daily 文件或 manifest.json，一直都是直接读一份手动拷贝来的数据库文件；`SqliteMerger`（合并算法）在代码里也一直没有被调用过。第6节描述的整套方案被确认为**从未真正投入使用的历史设计**，相关代码（`RunMergeAsync`/`SqliteMerger`/`FileNaming`/Fetcher界面的"合并"按钮及 `Manifest` 里的 `CurrentMaster`/`PreviousMaster`/`DailyFiles` 字段）已删除。第6节整节保留仅作为设计过程记录，不代表现状——现状就是上面这张图：Fetcher 产出 `current.sqlite`，用户手动拷贝一份给 Analyzer 读取（文件名约定为 `total.sqlite`），没有网盘环节。

## 3. 数据范围与数据分级

### 3.1 Level-1 与 Level-2 的区别（决定了免费接口能拿到什么）

| | Level-1（免费/低成本，现有东方财富接口属于此级） | Level-2（付费，通常按年订阅） |
|---|---|---|
| 盘口深度 | 五档买卖盘 | 十档买卖盘 |
| 逐笔成交 | 无，或只有"最近若干笔"抽样 | 完整逐笔成交，含买卖方向、毫秒级时间戳 |
| 逐笔委托（含撤单） | 无 | 有，含匿名委托序号 |
| 大单统计 | 无 | 有（或可从逐笔数据自行加工） |
| 基础报价/K线/成交量 | 有 | 有 |

**关键结论：即使是付费 Level-2 数据，也无法得知真实的"人"是谁在买卖**——这是交易所监管层面禁止对外披露的隐私信息。付费数据最多能给到：
1. 每笔成交的买卖方向（主动买/主动卖，基于成交价相对买卖一价推断）
2. 逐笔委托的匿名交易所内部序号（不对应真实身份）
3. 龙虎榜数据（仅限触发异常波动的个股，精确到"某证券营业部"买卖了多少，机构级别，非个人）

### 3.2 本期目标：只做 Level-1，Level-2 表结构也不预留

本期设计只覆盖 Level-1 数据，**不为 Level-2（完整逐笔成交、十档盘口）预留表结构**，原因：
- 免费公开接口（东方财富等）不提供3年历史的逐笔/十档数据，需要接入付费数据源才能拿到
- 第5节的容量评估表明，全市场多年逐笔数据的量级（百亿~千亿行）已经超出 SQLite 的合理使用范围，很可能需要换用列式/时序数据库（如 DuckDB、ClickHouse）才合适
- 既然存储引擎大概率不同，现在预留的表结构到时候也基本用不上，不如等真正确定要接入 Level-2 数据源、选定了存储方案之后再单独设计，避免预留的设计和实际需求对不上

### 3.3 数据类型

**交易数据（不可变时序事实）**：
- 日线、周线、月线（周/月建议由日线本地聚合计算得出，不单独请求接口，好处：离线也能算，不依赖数据源是否支持）
- 分时（1/5/15/30/60分钟）—— 免费接口历史深度有限（通常几个月到一年，非3年），需要抓取时实测确认
- 逐笔成交（谁在几时以什么价买/卖多少股）—— **本期不做，不预留表结构**，见3.2

**基本面/估值数据（会变化，需保留历史而非覆盖）**：
- 财务指标：营收、净利润、ROE、每股收益(EPS)、每股净资产(BVPS) 等，按报告期（季度/年度）存储
- 估值指标：PE、PB，按交易日存储（因为价格每天变，PE/PB也每天变）
- 数据源细节（东财具体接口字段）需要单独调研确认

### 3.4 多数据源（已实现）：东方财富 + 腾讯财经 + 新浪财经，单次运行只用一个源，可手动切换

单一数据源在批量抓取全市场股票时容易被对方的反爬/限流机制封锁IP（实测踩过坑）。为此支持接入**多个可选数据源**，但**同一次抓取运行只使用一个源，不自动混用、不自动切换**——原因是不同数据源尽管核心行情数据一致（见3.4末尾的实测校验），但不排除某些衍生字段的计算细节有差异，同一批数据里混用多个源会让"这批数据到底是谁算的"变得含糊，不如每次运行都明确只用一个源，数据口径统一、可追溯。

- `IBarDataFetcher` 接口不变，`TencentBarFetcher`（`web.ifzq.gtimg.cn` 的公开前复权K线接口）、`SinaBarFetcher`（`money.finance.sina.com.cn` 的公开经典K线接口，2026-07-08新增）分别跟 `EastMoneyBarFetcher` 并列，三者通过 `NamedBarSource`（名字+实现）注册成一个可选列表
- **抓取程序界面上有一个数据源下拉框**，用户先选一个源再点击"开始抓取"；如果当前选的源连不上/被限流，**手动切换下拉框到另一个源，重新点击"开始抓取"即可**——不需要额外操作，因为增量判断是按"每只股票最后已抓到哪天"来的，切换源之后重新运行会自动只补上一次没抓成功的那些股票，不会重复抓已经有的数据
- 东方财富、腾讯财经两个源经过实测校验：用贵州茅台 2026-06-25 实际分红事件比对，前复权日线数据逐日逐位完全一致（见项目讨论记录），确认切换数据源不会引入数据不一致的问题
- **腾讯接口有一个已知限制**：单次请求服务端硬性限制最多返回640条K线（无论请求参数怎么写），`TencentBarFetcher` 内部通过"按结束日期向前翻页、拼接结果"的方式自动处理，对调用方透明
- **`SinaBarFetcher`（2026-07-08新增，添加原因见下方"新增新浪K线源"）已知限制**：
  - 接口本身没有 beg/end 范围参数，只能"要最近N条"，`SinaBarFetcher` 按请求的日期区间估算一个足够大的条数一次性要回来再本地按日期过滤——实测这个接口没有明显的硬性条数上限（一次要8000条时，如实返回了贵州茅台从2001年上市至今的全部5956条历史，没有被截断/分页限制），不需要像腾讯那样分页拼接
  - 不直接返回成交额/换手率，这两个字段固定写0——已用 grep 确认全代码库没有任何筛选规则读取 `Bar.Amount`/`Bar.Turnover`（只有周/月线聚合求和、和界面展示用到），不影响任何条件判断，纯粹是这两项统计数字在界面上会显示为0
  - **是否前复权尚未验证**：东方财富(fqt=1)/腾讯(qfq)已确认互相一致（前一条），但新浪这个接口是否同样做了前复权、还是给的除权前原始价格，验证时东方财富/腾讯当时都连不上（跟这个源本身要解决的"东方财富连不上"问题巧合地同时发生），没能对着已知的贵州茅台分红事件做逐日对比。如果新浪给的是未复权数据，遇到分红/送股的股票，MA/MACD等技术指标会在除权日附近出现新浪跟东方财富/腾讯不一样的走势（只影响用新浪抓的那一段，不会追溯改动已经用别的源抓到的历史数据）。建议先用一两只近期有分红的股票试抓，肉眼对比K线图在除权日附近是否平滑，再决定要不要长期依赖这个源
- 以后要加新数据源（比如网易财经的新接口，AKShare 本身是 Python 库不能直接从 C# 调用，但可以参考它聚合的底层数据源自己实现对应的 HTTP 请求），只需要新写一个 `IBarDataFetcher` 实现，加入数据源列表即可，不需要改动抓取编排逻辑，下拉框里自动多一个选项

**新增新浪K线源的原因（2026-07-08）**：用户反馈东方财富在实际使用环境里基本连不上（同一个问题也影响了流通市值/资金净流入，见3.5/3.7节），并提出希望"能从一个源取数据，就不要从多个地方拼凑数据"——当时资金净流入已经默认走新浪，K线本身只有东方财富/腾讯两个选项，新浪缺席。加上新浪的K线接口经确认可用（覆盖沪深北三个交易所、历史深度充足）之后，把它也注册成第三个K线源。（流通市值同一天晚些时候也从腾讯换成了新浪，见3.5节最新状态——所以现在如果用户只信得过新浪，K线+流通市值+资金净流入三块都能落在同一家。）

**"Tencent"选项改成腾讯优先+新浪备用（2026-07-08，同一天再晚一点）**：用户实际用下来反馈新浪的抓取稳定性不如腾讯，所以不应该是两个平级选项——`TencentThenSinaBarFetcher`（`src/StockPlatform.Data/Remote/`）包一层：先试腾讯，只有某只股票腾讯拿不到时才回退到新浪重试这一只，不是整批都发两遍请求。这个回退只发生在"Tencent"这一个下拉框选项背后，独立的"Sina"选项（纯新浪、无回退）和"EastMoney"选项都不受影响，仍然是各自单一的数据源。已用一个会强制腾讯失败的测试验证过：腾讯失败时确实会自动改用新浪重试，日志里能看到"腾讯获取 xxx 失败...，改用新浪重试"这行提示。

### 3.5 流通市值抓取（已实现）：给分析程序"彬哥法"用

分析程序的"彬哥法"（见 [analysis-app-design.md](analysis-app-design.md) 3.2.4节）需要流通市值，`FundamentalMetric` 表本来就设计好了但一直没有数据获取程序往里写——现在补上。抽了一个 `IMarketCapFetcher` 接口（`StockPlatform.Logic.Abstractions`，`MarketCapEntry` 记录同样挪到了 `StockPlatform.Logic.Models`，方便这个接口和 `INetInflowFetcher` 一样跨项目引用），有两个实现：

- `EastMoneyMarketCapFetcher`（`src/StockPlatform.Data/Remote/`）：最初的实现，逐只股票查东方财富的单股行情接口（`push2.eastmoney.com/api/qt/stock/get?secid={市场前缀}.{代码}&fields=f116,f117`，`secid` 复用 `EastMoneyBarFetcher.ToSecId`）——`f117` = 流通市值，`f116` = 总市值，单位都是原始"元"（已用贵州茅台 600519 实测校验：约1.48万亿元，量级正确）
  - 最初是用东方财富的批量列表接口（`push2.eastmoney.com/api/qt/clist/get`，一次性拿全市场）实现的，上线后发现实测不可靠——反复出现连不上（HTTP连接被重置）、返回的JSON在固定字节数处被截断、或者返回200但所有股票的 `f117` 都是字符串占位符`"-"`而不是数字这三种情况，导致实际抓回的数据经常是0条。改成跟K线一样逐只股票查询之后，同样的股票live测试变成100%成功
  - 已知的一个坑：EastMoney 对少数代码（比如没有实时行情但代码结构合法的）返回的 `f117` 不是`"-"`占位符，而是字面量数字 `0`——代码里会把 `<=0` 的值当成"没有数据"一并跳过，不会把假的0写进数据库
  - **目前代码里默认没有用这个实现**——见下方状态变更记录
- `TencentMarketCapFetcher`（`src/StockPlatform.Data/Remote/`）：逐只股票查腾讯的公开实时行情接口（`qt.gtimg.cn/q={sh/sz/bj}{代码}`，前缀复用 `MarketClassifier.TencentSymbolPrefix`），回包是一个用`~`分隔字段的字符串（如 `v_sh600519="1~贵州茅台~600519~...~14992.23~14992.23~6.44~...";`）。**字段位置在实现前用3只不同特征的股票做过校验**：600519（贵州茅台，流通股本≈全部股本）和000001（平安银行，接近全流通）在候选字段位置两个数值相等/接近相等，300750（宁德时代，有明显限售股）两个数值明显不同（15367.90亿 vs 16702.21亿，流通&lt;总）——正好符合"这两个位置分别是流通市值、总市值"的预期模式，确认字段下标（0-index）44=流通市值（亿元）、45=总市值（亿元）后才写进代码
  - 回包声明 `charset=gbk`，需要先注册 `CodePagesEncodingProvider` 并显式按GBK解码字节，再按`~`切分字符串（不能对原始字节直接按`~`字节切分——GBK双字节编码的尾字节范围包含`~`对应的字节值，理论上可能把股票名称里的某个汉字字节误判成分隔符，导致后面的字段全部错位）
  - **目前代码里默认也没有用这个实现**——同样见下方状态变更记录
- `SinaListMarketCapFetcher`（`src/StockPlatform.Data/Remote/`）：**当前默认使用的实现**。不发单独的请求——`SinaStockListProvider` 扫全市场股票列表时（`Market_Center.getHQNodeData`，本来就是为了拿股票代码/名称）返回的每条记录里，本来就免费带了 `nmc`（流通市值，单位万元）和 `mktcap`（总市值，单位万元）两个字段，`SinaStockListProvider` 现在会把 `nmc` 一并解析进 `StockListEntry.CirculatingMarketCap`（`EastMoneyStockListProvider` 的列表接口没有这个字段，对应位置留 `null`），`SinaListMarketCapFetcher` 只是重新跑一遍这个列表扫描、按需要的代码过滤出市值。**字段含义在改用前用贵州茅台/平安银行/宁德时代三只股票交叉验证过跟 `TencentMarketCapFetcher` 抓到的值完全对得上**（小数点后几位的差异是两次请求之间几秒钟实时行情变化导致的，不是解析错误）
  - 效果：原来"全市场约5000只股票逐只发请求"（`RateLimiter` 限流下要跑很久）变成"约55页的股票列表扫描"（实测约23秒跑完全市场，加上后来补的翻页延时后会略慢一点，见下一条），既更快也更符合"一个源就够、不要多处拼凑"的偏好——`SinaStockListProvider` 本来就已经在别处被依赖（配对Tencent的K线源、以及新增的Sina K线源自己的列表步骤，见3.4节）
- **反爬限流的实测教训**：`SinaStockListProvider.GetAllStocksAsync` 原来翻页请求之间没有任何延时，只有失败之后的重试延时——开发这个功能时用脚本连续密集请求约80页，很快就收到了HTML反爬拦截页而不是JSON（不是理论推测，是实测踩到的坑）。既然这个类现在被 `SinaListMarketCapFetcher` 提到"每次拉取全部/拉取当天都要完整跑一遍"的更高频率，专门给翻页之间加了300毫秒的固定延时（`SinaStockListProvider.DelayBetweenPages`），降低触发限流的概率

**状态变更记录（2026-07-08）**：先是 `EastMoneyMarketCapFetcher`（live curl测试从开发环境是通的），后来发现用户实际使用环境里东方财富基本连不上（跟"耀哥法"资金净流入规则那次是同一个问题，见3.7节），换成 `TencentMarketCapFetcher`（逐只查询，验证有效）。当天晚些时候在验证新浪K线接口时，顺手发现新浪的股票列表接口本身就带流通市值字段，于是又换成了 `SinaListMarketCapFetcher`——`EastMoneyMarketCapFetcher`/`TencentMarketCapFetcher` 都实现了同一个 `IMarketCapFetcher` 接口，保留在代码里但目前都没有被任何composition root构造/使用。

- **固定用当前默认实现，不受用户选的K线数据源影响**：这块目前还没有像K线那样的用户可选下拉框，市值这个数值也不像K线行情那样需要"数据源一致性"的顾虑，直接固定用一个实现最简单
- **"拉取当天"现在也会完整扫一遍全市场股票列表**（只为了刷新市值——K线本身仍然只抓本地已知的股票，不重新扫描）——这是切到 `SinaListMarketCapFetcher` 之后带来的副作用，因为流通市值现在只能通过完整的列表扫描获得，不能像 `TencentMarketCapFetcher` 那样只查"拉取当天"要用到的那几个代码。用户已确认接受这个额外耗时（2026-07-08）：与其让"拉取当天"这种快速操作跳过市值更新、留着一份越来越旧的市值数据，不如统一每次都刷新，慢一点没关系

### 3.6 中标/订单公告抓取（已实现）：市场维度的公告全文检索

跟K线/市值这种"按股票逐个抓"不一样，中标/订单公告是**市场维度的关键词搜索**——不针对某只股票，而是全市场按关键词（比如"中标"、"签订合同"）搜公告，见 `AnnouncementFetchOrchestrator`：

1. **发现**：`CninfoAnnouncementSearchProvider`（巨潮资讯网全文检索）按关键词+日期窗口搜出全市场命中的公告标题/发布日期/PDF链接
2. **详情**：`EastMoneyAnnouncementDetailFetcher` 尝试按股票代码+标题+日期从东方财富公告库里找到对应文章拿正文纯文本（尽量避免下载PDF自己解析），找不到就只存标题/日期/PDF链接（软失败，不算这条公告抓取失败）
3. **提取**：`OrderWinExtractor` 从正文里找"合计"这类汇总行提取总金额，找不到就是 null，不会自己把明细行加总（防止误算）
4. **存储**：`OrderWinAnnouncement` 表，主键 `(code, title, publish_date)` 去重

**状态变更记录（2026-07-08）**：最初界面上是一个独立的"抓取中标/订单公告"按钮，需要用户自己填关键词和起止日期手动触发。现在整合进"拉取全部"和"拉取当天"里自动跑，不再单独触发（跟流通市值现在的地位一样——都是"这两个主操作顺带做的事"，不是用户要单独记得点的第三件事）：
- 关键词依然是界面上可编辑的设置（默认"中标,签订合同"，逗号分隔，清空则不抓），只是不再挂一个专属的触发按钮
- 日期窗口由调用方决定：**"拉取全部"用固定回看窗口**（`AnnouncementLookbackDaysForFetchAll`=14天到今天）——公告没有像K线那样按股票记录"上次抓到哪天"的水位线，重复扫描同一窗口靠主键去重是安全的，14天的重叠窗口足够覆盖两次"拉取全部"之间的正常间隔，不需要真的实现增量水位线；**"拉取当天"就只查那一天**，跟它对K线的语义一致
- 失败按非致命处理——记一条日志继续，不影响K线抓取的其余部分
- `FetchOrchestrator` 在"拉取全部"和"拉取当天"两个入口都会调用一次（`FetchMarketCapAsync`），失败按非致命处理——记一条日志继续，不影响本轮K线抓取
- 写入用 `IFundamentalMetricRepository.Upsert`（主键是`(code, metric_key, as_of_date)`的`INSERT OR REPLACE`）——同一天内如果跑了不止一次，只认最后一次抓到的值，不是"当天已经有了就跳过"。这是从最初的 `InsertOrIgnore`（谁先写谁留）改过来的：用户的实际用法是固定收盘后才拉取（此时价格已定，市值不会再变），但如果某天碰巧盘中先跑过一次、收盘后又跑一次，`InsertOrIgnore` 会把不够准的盘中值锁死、收盘后的准确值反而进不去，所以改成"后写的覆盖先写的"

### 3.7 主力净流入抓取（已实现，2026-07-08）：给分析程序"耀哥法"新增的"最近三天资金净流入"规则用

跟流通市值一样是逐只股票查询，但是**完全不同的数据集/接口**，也**不是** `FundamentalMetric` 快照表能装的——这是一个真正需要按天存历史的时间序列（要判断"最近三天"），所以单独建了 `NetInflow` 表（见第4节），跟 `Bar` 一样按"每只股票自己的水位线"增量抓取。抽了一个 `INetInflowFetcher` 接口（`StockPlatform.Logic.Abstractions`），有两个实现：

- `EastMoneyNetInflowFetcher`（`src/StockPlatform.Data/Remote/`）：最初的实现，查东方财富的个股资金流向历史接口（`push2his.eastmoney.com/api/qt/stock/fflow/daykline/get?secid={市场前缀}.{代码}&fields2=f51,f52&klt=101&beg=...&end=...`，`secid` 复用 `EastMoneyBarFetcher.ToSecId`）——`f51`=日期，`f52`=主力净流入净额（元）。字段含义在实现前用live数据做过恒等式校验：请求完整的 `f51~f65` 时，`f52`（主力净流入）恒等于 `f55`（大单净流入）+`f56`（超大单净流入），四个单量级分类（小/中/大/超大单）净额之和恒为0（符合"净流入是同一批成交按单量级重新分类"的定义），确认字段映射无误后才只请求 `f51,f52` 两个字段。**跟K线用的接口是两码事**：`fflow/daykline/get`（真正的多日历史，`beg`/`end` 参数生效，实测能拿到半年以上的数据）不同于 `fflow/kline/get`（这个接口只返回当天一条，`lmt` 参数不生效，容易被误认为是历史接口——已实测确认两者区别）。**目前代码里默认没有用这个实现**——见下方状态变更记录。
- `SinaNetInflowFetcher`（`src/StockPlatform.Data/Remote/`）：**当前默认使用的实现**。查新浪财经的个股资金流向趋势接口（`vip.stock.finance.sina.com.cn/quotes_service/api/json_v2.php/MoneyFlow.ssl_qsfx_zjlrqs?daima={sh/sz/bj}{代码}`，`daima`前缀复用 `MarketClassifier.TencentSymbolPrefix`——新浪和腾讯共用同一套sh/sz/bj前缀老约定，跟东方财富的数字前缀不是一回事）。实测确认这是一个真正的全历史接口（贵州茅台一次请求返回3964条，回溯到2010-03-01），但**没有beg/end/lmt这类范围参数**——请求日期区间是拿到完整响应后在代码里过滤的，不是发给服务端的，所以每次请求下载的数据量比东方财富那版大得多（单只股票超过1MB），全市场跑一遍的网络开销明显增加，但换来的是这个接口在实际环境里真的能连上。取的字段是 `netamount`（全部单量级——超大单+大单+中单+小单——净流入之和），跟东方财富的"主力净流入"（只算大单+超大单）定义不完全一样，因为新浪这个历史接口只单独暴露了 `r0_net`（超大单），没有大单的历史明细，凑不出跟东方财富一样口径的"主力净流入"；`netamount` 是对用户原话"资金净流入"更直白的理解（不特指"主力"）。

**状态变更记录（2026-07-08）**：最初实现时用的是 `EastMoneyNetInflowFetcher`（东方财富的资金流向历史接口），live curl测试从这个开发环境是通的。但用户反馈：他们实际使用环境里东方财富基本连不上，一直在用的是腾讯（K线）+新浪（股票列表），这套系统现在实际能用的数据源本来就不是东方财富。为此新增了 `SinaNetInflowFetcher` 并改成默认使用它，`EastMoneyNetInflowFetcher` 保留在代码里（实现了同一个 `INetInflowFetcher` 接口）但目前没有被任何composition root构造/使用，作为"东方财富哪天恢复可用，或者以后要给这块也做一个像K线那样的多数据源下拉框"时的备用实现。

- 两个实现都是独立的 `RateLimiter`，跟K线/市值互相独立，`FetchOrchestrator` 在"拉取全部"和"拉取当天"两个入口都会调用一次（`FetchNetInflowAsync`），失败按非致命处理——单只股票查不到就跳过，不影响K线抓取
- **固定用当前默认实现，不受用户选的K线数据源影响**（跟流通市值同样的理由——这块目前还没有像K线那样的用户可选下拉框）
- 写入用 `InsertOrIgnore`（主键 `code+period_start`）——已经抓到的历史交易日的资金净流入是既定事实，不像流通市值那样"同一天内后一次覆盖前一次"，重复抓同一天是无操作

### 3.8 Fetcher程序的可靠性/可运维性改进（已实现，2026-07-08）

一批跟"抓什么数据"无关、但直接影响这个程序好不好用/好不好排查问题的改动：

**失败股票记录+重新拉取**：K线抓取失败的股票代码会持久化记录在 `manifest.json` 的新字段 `Manifest.FailedCodes` 里（`FetchOrchestrator.ComputeUpdatedFailedCodes`）——每次"拉取全部"/"拉取当天"/"重新拉取失败股票"结束后都会更新：本轮尝试过且这次成功了的代码移出名单，这次还失败的代码加入/保留在名单里，没被本轮碰到的代码保持不动。界面上新增"重新拉取失败股票（N）"按钮（`FetchOrchestrator.RunRetryFailedAsync`），只针对这份名单里的代码重试K线——不重新扫描全市场列表、不重新跑公告（公告本来就是全市场批量扫描，不按股票记录失败，下次跑"拉取全部"/"拉取当天"自然会覆盖到）。可以反复点击：只要重试完还有股票失败，名单就不会清空，可以一直点到清零为止。用一个会模拟部分股票失败、再让它们分批恢复的测试验证过整个"记录失败→定向重试→名单收敛到0"的流程。

**状态变更记录（2026-07-09）**：最初"失败追踪+重试"只覆盖K线，用户反馈市值/资金净流入这两项也会失败，也应该能重试。扩展成三份独立的失败名单（`Manifest.FailedCodes`/`FailedMarketCapCodes`/`FailedNetInflowCodes`），"重新拉取失败股票"按钮改成三项都会分别处理，界面数字是三份名单之和。两项的失败粒度不一样：
- 资金净流入是逐只单独查的（跟K线一样），能精确到具体哪只失败，重试也是只查失败的那些——用测试验证过：3只股票里1只资金净流入失败，重试后接口只被多调用了1次（不是重新查全部3只）
- 流通市值现在走的是"新浪股票列表整体扫一遍"（`SinaListMarketCapFetcher`），不是逐只单独查，所以"失败"粒度是"这一整轮扫描失败了"——扫描失败就把这一轮请求的代码全部记为失败，扫描成功就全部移出（某只股票本来没有市值数据不算失败）。这意味着**市值的"重试"本质上还是要整轮重新扫描**，跟直接再跑一次"拉取全部"里的市值步骤没有本质区别，只是能正确清零失败名单——不像K线/资金净流入那样可以真正做到"只查失败的那几只，省时间"

**顺带修的一个坑（已随2026-07-09的合并功能移除而失效，见2节状态变更记录）**：验证"重新拉取失败股票"时发现，如果同一天内跑过不止一次会产出同名日增量文件的操作（比如"拉取全部"之后紧接着"重新拉取失败股票"），`ProduceOutputFileAsync` 原来的"先删除旧文件再重新创建"逻辑会因为 Microsoft.Data.Sqlite 的连接池还占着上一次写入用的文件句柄而报"文件正在被占用"——这是一个实测复现出来的真实bug，不是假设性的，当时的修法是删除前先调用 `SqliteConnection.ClearAllPools()`。这个日增量文件生产的流程本身已经整体删除，这个坑随之一起消失，记录在这里只是留个痕迹。

**关闭前的两道保护**（`MainWindow.xaml.cs`）：正在抓取时，直接拦截关闭并提示"请先点击停止"（`Window.Closing` 里检查 `MainViewModel.IsBusy`，是则 `e.Cancel = true`，不弹二次确认，因为唯一正确的操作就是先停止）；不在抓取时，也要求用户在一个"确定要关闭程序吗？"的确认框里点"是"才真正关闭，防止误触关闭按钮/Alt+F4丢失当前的运行日志或状态。

**运行日志落盘**（`FetchPaths.LogFilePath` = `data/local/fetch.log`）：`MainViewModel` 在构造时打开一个 `StreamWriter(append: false){AutoFlush = true}`，每次写入界面日志（`Log()`方法）时同步写一份到这个文件——每次程序启动清空重写（不是按天滚动/长期保留，这只是给"程序意外退出时还能看到发生了什么"用的诊断日志，不是审计记录），`AutoFlush` 保证每行一写完就落盘，即使程序崩溃/被强制结束，打开这个文件也能看到崩溃前最后发生的事，不需要依赖还开着的界面窗口。

### 3.9 收盘后数据免重复抓取（已实现，2026-07-09）

**用户提出的问题**：现在增量抓取只有"拉取全部"（水位线增量）和"拉取当天"（指定单日，忽略水位线）两种模式；如果程序在收盘前（比如盘中）就抓过"今天"这一天，K线/资金净流入用的 `InsertOrIgnore` 会把这条不完整的记录永久锁住——哪怕后面再收盘后重新跑一次，"今天"已经有记录了，`InsertOrIgnore` 什么都不会做，错的/不完整的数据永远不会被修正。用户提出的解决方向：记录每一条数据实际抓取到的时间，用来判断这条数据是不是已经在收盘后确认过、以后不用再抓；这个"要不要重新抓"的判断不只按股票，还要按"股票的某项具体数据"（K线、资金净流入分别判断，因为它们是独立抓取的）。

**实现**：

- `Bar`/`NetInflow` 模型各加一个 `FetchedAt`（墙钟时间，不是交易日日期）字段，`Bar`/`NetInflow` 表各加一列 `fetched_at TEXT`——已建好的老数据库文件（这个字段之前不存在）靠 `SqliteSchema.EnsureSchema` 里新增的 `ALTER TABLE ... ADD COLUMN`（先用 `pragma_table_info` 检查列是否已存在，保持幂等可重复调用）自动补上，不需要用户手动迁移；老数据行读出来 `FetchedAt` 是 `DateTime.MinValue`（永远判定为"未确认"，直到重新抓到一次），不会误报成"已经是最终数据"。
- 判定逻辑是 `FetchOrchestrator.IsConfirmedFinal(fetchedAt, tradingDay) => fetchedAt >= tradingDay.Date.AddHours(16)`——最初用15:00（常规连续竞价收盘时间），后来改成16:00：15:00~15:30还有盘后定价交易（大宗/固定价格成交），这段时间抓到的数据不算真正确定，16:00才能确保盘后定价交易也结束了。故意不处理节假日前半天早收盘的少数情况（用固定较晚的时间点，最坏情况是多等一会儿才认定为最终，不会出现"提前认定、结果数据其实还会变"的反向错误）。
- 三个"拉取"入口分两种检查方式：
  - "拉取全部"/"重新拉取失败股票"（水位线增量）：查 `GetLatestBarInfo`/`GetLatestRowInfo`（新增方法，一次查询同时拿到最新日期+抓取时间）——如果最新日期正好是"今天"，才需要看是否 `IsConfirmedFinal`；早于今天的日期天然已经是最终数据，直接跳到下一天。
  - "拉取当天"（指定单日）：直接查这个具体日期本地是否已有记录、以及是否 `IsConfirmedFinal`——不看整体水位线在哪。
- 写入侧：`period_start` 等于当前日期的那一行走 `SqliteBarUpsert.Upsert`/`SqliteNetInflowRepository.Upsert`（`INSERT OR REPLACE`，覆盖旧值），其余日期仍然是 `InsertOrIgnore`（历史交易日是既定事实，不会变，也不应该被覆盖）——一次抓取返回的多天数据里，只有"今天"这一天可能需要覆盖，其它天必然是第一次见到的新事实。
- 流通市值（`SinaListMarketCapFetcher`）不受影响——它已经是"每次都用最新的覆盖写入"，不参与这套水位线判断，本来就没有这个问题。
- 用两个脚本验证过：(1) 端到端场景——模拟10点抓到（未收盘）→ 16点再抓一次（应该重新发请求并覆盖）→ 再抓第三次（应该完全跳过，不发请求）；(2) 老数据库迁移——手工建一个没有 `fetched_at` 列的旧版Bar表、插入一行旧数据，跑 `EnsureSchema` 确认不报错、老数据能正常读出（`FetchedAt` 是 `MinValue`）、重复调用 `EnsureSchema` 也不报错、迁移后的新写入能正常工作。

## 4. 数据库表结构设计

设计原则：**用"维度值"代替"写死字段"**，保证后续加新的数据类型/周期时不需要改表结构、不需要数据库迁移。

```sql
-- 行情K线，多粒度统一存储
CREATE TABLE Bar (
    code TEXT NOT NULL,
    granularity TEXT NOT NULL,   -- 'day' / 'week' / 'month' / 'min1' / 'min5' / 'min15' / 'min30' / 'min60' / ...
    period_start TEXT NOT NULL,  -- ISO日期或日期时间，视granularity而定
    open REAL, close REAL, high REAL, low REAL,
    volume REAL, amount REAL, pct_chg REAL, turnover REAL,
    PRIMARY KEY (code, granularity, period_start)
);

-- 基本面/估值指标，Key-Value 式，加新指标不改表结构
CREATE TABLE FundamentalMetric (
    code TEXT NOT NULL,
    metric_key TEXT NOT NULL,    -- 'revenue' / 'net_profit' / 'roe' / 'eps' / 'bvps' / 'pe' / 'pb' / ...（未来可加新值，无需改表）
    as_of_date TEXT NOT NULL,    -- 财务指标=报告期，估值指标=交易日
    value REAL,
    source TEXT,
    fetched_at TEXT,
    PRIMARY KEY (code, metric_key, as_of_date)
);

-- 主力净流入（大单+超大单净流入之和，元），按天存的独立时间序列，不是FundamentalMetric能装的快照
CREATE TABLE NetInflow (
    code TEXT NOT NULL,
    period_start TEXT NOT NULL,  -- 交易日
    main_net_inflow REAL,
    PRIMARY KEY (code, period_start)
);

-- 股票基础信息
CREATE TABLE StockMeta (
    code TEXT PRIMARY KEY,
    name TEXT,
    exchange TEXT,
    list_date TEXT,
    last_updated TEXT
);
```

代码层面的接口预留（延续现有 Logic/Data 分层思路）：
- `IBarDataProvider`：查/写 Bar，参数带 `granularity`，加新粒度不改接口签名
- `IFundamentalMetricProvider`：按 `metric_key` 通用读写，加新指标不改接口

（不定义 `ITickTradeProvider`——Level-2 逐笔数据本期不做，见3.2，等真正要做时再连同存储引擎一起设计。）

## 6. 无服务器多机共享方案（历史设计，已废弃，见2节状态变更记录）

> **本节整节都是历史设计记录，不代表现状**：下面描述的 master/daily 文件、`manifest.json`、"合并"操作、百度网盘上传/下载，实际都没有被分析程序消费过（分析程序一直只读手动拷贝来的单一数据库文件），相关代码已于2026-07-09删除。保留本节是为了留存当时的设计过程和调研结论（尤其6.5.1节百度网盘API调研的结论仍有参考价值），不需要按本节内容理解当前实现。

约束：没有服务器，需要多人/多机共享同一份数据。核心原则：**SQLite 不适合在网络共享盘上被多方同时写入**，但"单一生产者写 + 多个消费者只读"是安全的。

### 6.1 文件角色

```
共享目录/
  stockdata_master_20260701.sqlite   ← 当前总数据文件（最新）
  stockdata_master_20260601.sqlite   ← 上一个总数据文件（备份，最多保留这1个）
  daily/
    stockdata_daily_20260702.sqlite  ← 每日增量，只保留最近7天
    stockdata_daily_20260703.sqlite
    ...
  manifest.json                       ← 描述当前状态，供消费者知道读哪些文件
```

`manifest.json` 结构：
```json
{
  "currentMaster": { "file": "stockdata_master_20260701.sqlite", "asOfDate": "2026-07-01" },
  "previousMaster": { "file": "stockdata_master_20260601.sqlite", "asOfDate": "2026-06-01" },
  "dailyFiles": [
    { "file": "stockdata_daily_20260702.sqlite", "date": "2026-07-02" },
    { "file": "stockdata_daily_20260703.sqlite", "date": "2026-07-03" }
  ]
}
```

### 6.2 生产者（数据获取程序）职责

数据获取程序对外提供两种独立的操作，各自都遵循"本地先做完 → 提示上传"的两步流程（见6.6）：

**操作一：抓取**
1. 首次运行：产出 `stockdata_master_{日期}.sqlite`
2. 之后每天：只产出当天的 `stockdata_daily_{日期}.sqlite`（增量，只含当天新增/变化的数据）
3. 每次产出文件后更新本地的 `manifest.json`

**操作二：合并**（不设自动触发条件，纯粹由**用户点击"合并"按钮**触发，想合并的时候自己点）
1. 把当前 master 文件 + 目前已下载到本地的**所有** daily 文件，用第6.4节的合并逻辑整合到一起
2. 按命名规则生成**新的** master 文件（文件名用合并当天的日期，如 `stockdata_master_20260801.sqlite`），旧 master 自动降级为"上一个"，更老的按第6.2节末尾的清理策略删除
3. 更新本地 `manifest.json`：`currentMaster` 指向新文件，`previousMaster` 指向刚才降级的旧文件，已被合并进去的 daily 文件从 `dailyFiles` 列表移除
4. 合并完成后，跟抓取操作一样，走"提示是否上传到网盘"的流程（见6.6），把新的 master 文件和更新后的 manifest 推到网盘，同时可以把网盘上已经被合并掉的旧 daily 文件一并清理

清理策略：
   - master 只留最新 + 前一个，超过2个删最老的
   - daily 只留最近7天，超过的删除

### 6.3 消费者（每台机器的分析程序）职责

每台机器本地维护一份**已经物理合并好的独立数据库** + 一个同步状态文件：

```json
// local_sync_state.json（消费者本地文件）
{
  "basedOnMaster": "stockdata_master_20260701.sqlite",
  "dataAsOfDate": "2026-07-05",
  "appliedDailyFiles": ["stockdata_daily_20260702.sqlite", "stockdata_daily_20260703.sqlite"]
}
```

**首次在某台机器运行**（无本地状态文件）：
1. 读共享目录 manifest，取**当前最新**的 master 文件，复制到本地作为起点
2. 根据该 master 的 `asOfDate`，找出共享目录里比这个日期新的所有 daily 文件
3. 依次合并进本地数据库，更新本地状态

**之后每次运行**：
1. 读本地状态的 `dataAsOfDate`
2. 只拉共享目录里"日期比本地更新"的 daily 文件，合并进本地库，更新状态
3. 不重新拉 master，除非发生下面的断档情况

**断档情况**（本地太久没运行，本地 `dataAsOfDate` 和共享目录现存最早的 daily 文件之间有缺口，说明中间的 daily 已被清理）：
- 退化为"首次运行"流程：重新拉最新 master + 该 master 之后的 daily，本地状态重置

### 6.4 合并操作实现方式

无论是生产者合并 daily 到 master，还是消费者合并 daily 到本地库，动作一致：

```sql
ATTACH DATABASE 'stockdata_daily_20260702.sqlite' AS delta;
BEGIN;
INSERT OR IGNORE INTO main.Bar SELECT * FROM delta.Bar WHERE granularity NOT IN ('week','month');
INSERT OR REPLACE INTO main.Bar SELECT * FROM delta.Bar WHERE granularity IN ('week','month');
INSERT OR IGNORE INTO main.FundamentalMetric SELECT * FROM delta.FundamentalMetric;
COMMIT;
DETACH DATABASE delta;
```

- `day`（以及未来任何直接抓取而非派生计算的粒度，如分钟线）是不可变的原始事实，一旦写入就不会再变，用 `INSERT OR IGNORE` 防止主键冲突导致合并失败（正常情况下不会有重复，防御性处理）
- `week`/`month` 是派生数据：抓取程序每次抓取都会基于全部日线历史重新计算"当前尚未收盘"的那一周/那一月并覆盖本地（见第6.2节 `SqliteBarUpsert`），因此同一个 `(code, granularity, period_start)` 主键在更新的 daily 增量文件里可能带着比目标库里已有行更新的值。如果对 `week`/`month` 也用 `INSERT OR IGNORE`，目标库里的当前周/当前月会永远停留在**第一次**合并进来时的值，不会再随后续 daily 文件更新——因此必须用 `INSERT OR REPLACE`，让后合并的（更新的）daily 文件覆盖旧值。这依赖调用方按日期顺序依次合并 daily 文件（消费者侧 `SyncOrchestrator` 已经这样做），保证"后合并的值更新"
- 整个合并过程包在一个事务里，保证要么全部成功要么全部不生效，避免合并到一半崩溃导致本地数据库损坏

### 6.5 共享介质：网盘（已确定），当前用百度网盘 + 手工上传（暂无开发者凭证）

共享介质确定为**网盘**，不依赖公司内部共享盘/NAS。

**现状**：暂时没有申请百度网盘开放平台的开发者凭证（AppKey/SecretKey/access_token），所以**当前不做自动化的API上传/下载**，改成"程序只负责把文件命名准备好，上传这个动作由用户手工打开百度网盘网页完成"。这不影响后续升级成自动化——接口设计上仍然按可替换的抽象来定义，只是当前的"实现"是一个手工提示，而不是真正调用网盘API：

```csharp
public interface ICloudStorageClient
```

- 具体用哪个实现由配置决定（类似原 StockAnalyzer.Logic 里 AppConfig 的思路，该项目已删除，见第1节变更记录），换成自动化方案时只是切一下配置

**只要文件命名规则符合第6.1节约定（`stockdata_master_{日期}.sqlite` / `stockdata_daily_{日期}.sqlite` / `manifest.json`），用户手工上传到网盘哪个目录、什么时候上传，都不影响下游消费者读取——消费者只关心文件名和 `manifest.json` 里的内容，不关心它是怎么传上去的。**

**将来申请到开发者凭证后**（前置条件：百度网盘开放平台注册账号、创建应用拿到 AppKey/SecretKey、完成一次 OAuth 用户授权拿到 access_token/refresh_token），随时可以把 `ManualUploadPrompter` 换成真正的 `BaiduNetdiskClient`，不影响已经写好的其他逻辑。

1. **提交提取码验证**：

   ```
   Body（form-urlencoded）: pwd={提取码}
   Referer: https://pan.baidu.com/s/1{完整分享ID}
   ```
   其中 `surl` = 分享链接 `/s/1EDo57ELpCEGy-dZtIc87Lg` 里 `/s/1` 之后的部分，即 `EDo57ELpCEGy-dZtIc87Lg`（注意去掉那个前导的 `1`）。

2. **访问分享页面拿到根目录文件列表**（验证成功、cookie 生效之后）：

   返回的 HTML 里直接内嵌了一段 JSON（不需要另外调用API），可以用正则或字符串查找定位 `"file_list":[...]` 这段，每个条目包含 `fs_id`（文件唯一ID）、`isdir`（0=文件，1=文件夹）、`server_filename`（文件名）、`size`、`path` 等字段。也能拿到 `shareid` 和 `share_uk`（页面里 `yunData={...shareid:"...", share_uk:"..."}` 这段）。
   实测这个链接的根目录返回了一个文件夹 `stock-share`（`fs_id=806084441431740, isdir=1`）。

**后续调研（子文件夹列表已打通）**：

之前"子文件夹列不出来"的根因找到了——**不是签名问题，是漏了 `app_id` 参数**。真实网盘目录结构调整为直接把 master/daily/manifest 放在分享文件夹根目录下（`stock-share/` 直接含 `stockdata_master_*.sqlite`、`manifest.json`、`daily/` 子目录）之后，用下面的请求可以稳定拿到子目录内容（`errno:0`）：

```
GET https://pan.baidu.com/share/list?uk={share_uk}&shareid={shareid}&order=time&desc=1&showempty=0&web=1&page=1&num=100&dir={URL编码后的目录路径，如 /stock-share/daily}&t={毫秒时间戳}&channel=chunlei&clienttype=0&app_id=250528
```
- `shareid`、`share_uk`（请求里的参数名是 `uk`）从分享页面 HTML 内嵌的 `yunData={...shareid:"...", share_uk:"..."}` 里取
- `app_id=250528` 是关键——同样能在根目录 `file_list` JSON 的条目里看到这个字段（`"app_id":"250528"`），照抄即可，不用查文档
- 返回的 `list` 数组里每个条目就是 `fs_id`/`isdir`/`server_filename`/`size`/`md5`/`path`，跟根目录 `file_list` 的字段基本一致，分页用 `page`/`num`（本项目文件数量级用不到分页）

**结论：文件下载这条路走不通（不是参数没凑对，是百度网盘刻意要求登录）**——用拿到的真实 `fs_id` 反复测试 `https://pan.baidu.com/share/download`（POST，带 `fid_list`/`uk`/`shareid`/`primaryid`/`app_id` 等各种参数组合）和 `/api/sharedownload`，全部固定返回 `{"errno":112}`。做了对照实验排除"参数没配对"的可能：
- 换成明显无效的 `fid_list`（不存在的 fs_id）—— 同样是 `errno:112`，说明服务端根本没走到"校验这个文件"这一步
- 完全不带任何 cookie（连 verify 拿到的 `BDCLND` 都不带）发起请求 —— 结果还是 `errno:112`，跟带 cookie 时完全一样
- 分享页面 HTML 全文搜索不到任何 `sign`/`timestamp` 的现成值，且这两次对照测试的结果不随参数变化，说明缺的不是"算出一个签名"，而是这个接口本身要求一个**登录态**（真实账号的 `BDUSS`，即百度登录 cookie）才放行——这与开源资料里"分享链接客户端下载普遍要求 `BDUSS`"的说法一致（例如 AList 的百度分享驱动文档也明确写着下载需要 `BDUSS` 这个登录 cookie）。这是百度网盘刻意做的防盗链/反爬限制，不是随便调参能绕过的。

**结论落地**：匿名（不登录）方式**只能列目录、拿元数据（文件名/大小/md5/fs_id），拿不到真实文件字节**。要自动下载真实文件内容，唯一现实的路径是让用户从自己已登录的浏览器里手动导出 `BDUSS`（+通常还需要 `STOKEN`）这个登录态 cookie，粘贴到本地配置里，程序带着这个 cookie 去请求已登录用户视角下的下载接口（此时应该不再受 errno 112 限制）。这跟"申请官方开发者凭证（AppKey/SecretKey+OAuth）"是两条不同的路：前者是"借用你自己账号的网页登录态"，免申请、但 cookie 会过期需要定期更新、且相当于把账号登录凭证放在本地文件里，需要用户接受这个安全取舍；后者更规范但要走官方申请流程（见 6.5 节）。

**目前状态**：`ListFilesAsync` 已经有明确可用的实现方案（verify + 根目录/子目录 `share/list`，全部匿名可用），但 `DownloadAsync` 卡在"匿名做不到，需要登录态 cookie"这个根本限制上，需要用户决定要不要接受"手动提供 BDUSS cookie"这个方案再继续实现。

### 6.6 通用两步流程：本地先做完 → 提示手工上传

数据获取程序的**抓取**和**合并**这两种操作，都遵循同一个两步模式：

1. **第一步：本地操作**——抓取操作在本地生成/更新 daily（或首次的master）文件；合并操作在本地把 daily 整合进 master、按命名规则生成新 master 文件（见6.2）。这一步完全不涉及网络存储，失败了不影响已有的本地数据
2. **第二步：提示上传**——本地操作成功后，程序提示"数据已更新，请手动上传以下文件到网盘：{文件列表}"（当前是6.5节的手工过渡方案；以后接了真实API，这一步就变成用户确认后自动上传，交互上是同一个"确认/提示"位置，只是背后动作从"手工"换成"自动"）

这样设计的好处：无论上传是手工还是以后自动化，本地数据都已经是完整可用、命名正确的，不会因为上传环节（不管是人操作失误还是API失败）连累抓取/合并的结果本身。

### 6.7 抓取必须手动触发 + 限速策略

**手动触发**：数据获取程序不会自己定时/静默发起抓取，必须用户主动点击"开始抓取"（或类似的确认动作）才会真正调用东方财富接口。这一点跟第2节"分析程序下载数据也要点击才触发"是同一个原则——凡是会对外发起大量网络请求的动作，都要有一个明确的用户触发点，不做后台自动静默调用。

**限速策略**：为了避免请求过于频繁被数据源限流/封禁IP，抓取时对每个请求之间加统一的延时（具体延时值和并发数留到实现阶段调参，先按保守值起步，比如控制并发数在个位数、每次请求间隔几百毫秒量级，观察是否被限流再调整）。

这里有一个关键判断：**限速策略不需要区分"首次全量"和"之后每日增量"两种场景，统一用同一套保守限速就够了**：
- **首次全量**（全市场 × N年历史，N见下方）：数据量最大，本来就是"一次性慢工程"，跑几个小时甚至更久都可以接受，限速造成的额外耗时不是问题，反而是必须的（防止被封）
- **之后每日增量**：每次只抓"每只股票的最新一天"，数据量本身就很小，就算按同样保守的限速策略跑，实际耗时也很短（几分钟量级），不需要为了"跑快点"单独放开限速、增加被封风险

所以代码里不需要做"首次用慢速、之后用快速"这种模式切换，统一一套限速参数即可，简化实现，也更安全。

**状态变更记录（2026-07-08）——首次回看年数改成可调**：这个"N年"最初是写死的3年常量（`FetchOrchestrator` 里 `today.AddYears(-3)`），现在改成界面上"首次回看：__年"这个输入框，默认还是3，作为 `RunFetchAsync` 的参数传下去。注意这个设置**只影响本地完全没有历史的股票**（真正的首次运行，或者运行期间新出现在股票列表里的新股，比如新上市）——已经在本地有数据的股票，续抓的起点永远是"自己上次抓到的日期+1"，不受这个设置影响，调大或调小这个数字不会导致已有数据被重新抓一遍。

**同一次变更——界面新增"数据状态"提示**：Fetcher 主界面现在会显示本地数据的覆盖范围（`Bar` 表 day 粒度的最早/最晚 `period_start`）和上一次"拉取全部/拉取当天"实际完成的时间（`FetchOrchestrator.GetDataStatus()`，读 `Manifest.LastFetchAt`/`LastFetchKind`，每次抓取结束后更新，即使这次没抓到新数据也会更新——"检查过但没有新数据"本身也值得记录），让用户不用凭感觉判断"我今天到底抓过没有"，也不用打开数据库文件才能看到数据到底覆盖到哪天。

### 6.8 限速策略的三层防护（已实现）

在6.7节"并发+固定延时"的基础上，`RateLimiter`（见 [RateLimiter.cs](../src/StockPlatform.Data/Remote/RateLimiter.cs)）额外加了两层：

1. **失败重试**：最多重试2次，间隔固定 2秒、10秒
2. **全局熔断**：请求遇到 403/429/空响应/连接失败（统一归类为 `RateLimitedException`），会被判定为"疑似触发反爬限流"；**这次调用自己的2次重试用完、最终还是失败之后**，才触发一个全局暂停（5~15分钟随机，避免多个并发请求同时解除暂停造成瞬时冲击），暂停期间**其他后续请求**会先等待暂停结束再尝试
3. **请求超时**：统一设为12秒（10~15秒区间内），避免单个请求卡死占用并发名额

**踩过一个真实的坑，记录一下**：最初实现时，熔断触发的时机写错了——在"每一次"失败（包括还会重试的那几次）都触发熔断，导致这次调用自己的下一次重试会立刻等待自己刚设的5~15分钟暂停，把本该12秒完成的重试拖到了20多分钟。用模拟测试（不针对真实接口，避免真的被封）验证出这个问题后修正为：**熔断只在"这次调用的重试全部用完、即将放弃"的那一刻触发**，只影响之后的其他请求，不会拖慢这次调用自己的重试节奏。修正后重新测试，模拟"失败两次+第3次成功"确实耗时12秒（2s+10s），符合预期。
